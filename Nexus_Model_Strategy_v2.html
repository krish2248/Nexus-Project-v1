🎨 Nexus Design System — Reference (paste this into UI sessions)

  • Fonts: Syne 700/800 for all headings and brand text · DM Sans 400/500/600 for all body text · JetBrains Mono 400/500 for all code, labels, timestamps, badges. Load from Google Fonts.
  • Background: #080B14 primary · #0D1220 panels/sidebar · #111726 cards and hover states. Grid overlay: linear-gradient(rgba(255,255,255,0.018) 1px, transparent 1px) at 44px spacing.
  • Accent colors: Cyan #67E8F9 (primary actions, highlights) · Purple #A78BFA (secondary, music) · Green #34D399 (success, online) · Yellow #FCD34D (warnings, priority) · Red #F87171 (errors, urgent).
  • Ambient glow: Always add two radial gradient blobs in the background — cyan top-left, purple bottom-right, low opacity (~0.07). Pointer-events: none.
  • Cards: background: rgba(13,18,32,0.95) · border: 1px solid rgba(255,255,255,0.08) · border-radius: 14px · backdrop-filter: blur(20px). On hover: border brightens to 0.13 opacity, transform translateY(-2px).
  • Buttons: Primary = linear-gradient(130deg, #67E8F9, #A78BFA) with dark text, Syne font 700. Secondary = transparent with border. Ghost = no border, hover shows light bg.
  • Badges/Pills: Semi-transparent background using color at 0.10 opacity, same color border at 0.25 opacity, text in full color. Font: JetBrains Mono 10-11px.
  • Live clock: Use setInterval every second, new Date(), format as HH:MM AM/PM + full date string. Display in sidebar or header.
  • Animations: @keyframes fadeUp (opacity 0→1, translateY 14px→0) for page load · @keyframes glow for logo pulse · CSS transitions 0.18s ease on all interactive elements.
  • Scrollbar: 4-5px width, transparent track, rgba(255,255,255,0.1) thumb, 3px border-radius.
  • Reference files: See index.html (pre-lobby home) and lobby.html (full workspace) in the Nexus_v1.zip for complete production-ready examples to reference or extend.

The Golden Rules
Non-negotiable. Follow every rule on every session for 28 weeks.

Choose your model BEFORE opening a chat

Look up today's day in the table. Check the Model column. Open that model before typing a single word. Never open a chat and decide mid-session. The decision is made here, in this document.

OPUS = fresh 5-hour window only

The window resets from your first message. An Opus session with 8K tokens remaining will run out mid-debug. Schedule all Opus sessions as the very first task of your day — never at the end of a depleted window.

Never switch models mid-session on a hard task

If debugging a Yjs CRDT conflict or LiveKit connection issue with Opus, do NOT switch to Sonnet to save tokens. The diagnostic context built across messages is what makes Opus valuable. Losing it mid-debug costs more time than it saves.

One task per session

Each session focuses on one numbered task from the roadmap. Do not mix tasks. A session trying to implement DMs + @mentions + link previews will produce worse code than three focused sessions.

Paste design brief for ALL UI sessions

Always paste the Design Brief (new section above) at the start of any session building or modifying frontend. Without it Claude generates generic UI. With it: same Nexus design every time. This is the single most important new rule added to this guide.

Save Opus output as reference documents

Every Opus session produces a specification subsequent Sonnet sessions depend on. Save: ARCHITECTURE.md, SCHEMA.md, REALTIME_DESIGN.md. Paste the relevant section as context at the start of related Sonnet sessions.

Commit to GitHub after every session

Every session ends with git commit. Format: feat(phase1): implement invite system backend. Builds commit history, creates daily restore points, means you never lose more than one session.

Zero TypeScript errors before phase transitions

Run tsc --noEmit before tagging any phase complete. Type errors in Phase 1 compound into 10x harder bugs in Phase 3. The type system is your documentation — keep it clean.
