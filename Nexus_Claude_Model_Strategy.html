The Golden Rules
Non-negotiable principles for every session of the Nexus build.

Choose your model BEFORE opening a chat

Look up today's day in the table. Check the Model column. Open that model before typing a single word. Never open a chat and decide mid-session. The model decision is made here, in this document, not in the moment.

OPUS = Fresh 5-hour window only

The 5-hour window resets from your first message. An Opus session started with 8K tokens remaining will run out mid-debug or mid-architecture. Schedule ALL Opus sessions as the very first task of your day — never at the end of a depleted window.

Never switch models mid-session on a hard task

If you are debugging a Yjs CRDT conflict or a LiveKit connection issue with Opus, do NOT switch to Sonnet to save tokens. The diagnostic context built across messages is what makes Opus valuable. Losing that context mid-debug costs you more time than it saves in tokens.

One task per session

Each session focuses on one numbered task from the roadmap. Do not mix tasks. Claude works best with a focused, bounded scope. A session that tries to implement DMs + @mentions + link previews will produce worse code than three focused sessions.

Always paste your current file

Claude has no memory between sessions. When working on a specific file (schema, middleware, component), paste the current file content at the start of the session. Without it, Claude may generate code that conflicts with your existing implementation.

Save Opus output as reference documents

Every Opus architecture session produces a specification that subsequent Sonnet sessions depend on. Save the full Opus output: ARCHITECTURE.md, SCHEMA.md, REALTIME_DESIGN.md, etc. At the start of related Sonnet sessions, paste the relevant section as context.

Commit to GitHub after every session

Every session ends with a git commit. This builds your commit history, creates daily restore points, and means you never lose more than one session of work. Commit message format: feat(phase1): implement invite system backend

Zero TypeScript errors before moving phases

Run tsc --noEmit before tagging any phase complete. Type errors in Phase 1 compound into 10x harder bugs in Phase 3. The type system is your documentation — keep it clean.
